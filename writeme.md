1. **UI: вікно запуску та вікно процесу**
   1. Запуск коду має ініціювати відображення стартового UI-вікна. До явного натискання кнопки запуску жодні бекенд-операції не виконуються.
   2. Скіни, логіку, розгалуження (зокрема async для вікна процесу), а також роботу з tk (особливо перевірки звернень до потенційно неіснуючих об’єктів) допускається запозичування логіки з аналогічних .py файлів.
   3. Стартове вікно та вікно процесу — ніколи не центруємо по екрану. // Це не стосується messagebox-вікон помилок та повідомлень про успіх...

2. **BACKUP файлів запису**
BACKUP_ROOT = BASE_DIR / f"{folder/file_name} - {timestamp (datetime.now().strftime("%Y.%m.%d %H.%M"))}"
   Якщо код виконує запис у файл, перед будь-якою модифікацією: створюємо копію оригінального файлу, додаємо до назви копії timestamp, та переміщуємо копію до кошика. // Це дозволяє за потреби відновити оригінал або переглянути історію змін без накопичення зайвих копій у робочих директоріях.

3. **Tk**
   1. Якщо є вікно запуску або вікно процесу, усі messagebox із релевантними помилками та повідомленнями прив’язуємо саме до них, а не до прихованого root. Водночас окремі фінальні повідомлення/вікна про успішне виконання коду прив’язуємо до прихованого root, щоб на фоні не з’являлося ще одне вікно-привид.
   2. Під час знищення tk слід діяти безпечно. Особливу увагу приділяємо перевірці звернень до об’єктів tk, які на цей момент уже можуть не існувати. Ця проблема особливо помітна під час конвертації .py у .exe: коли кожна така помилка призводить до появи окремого вікна з повідомленням про помилку.

4. **PyInstaller / EXE**
   1. Для невеликих helper-функцій, які критичні для виконання конкретного скрипта, не створюємо окремий спільний модуль без явної потреби. Краще дублювати короткий локальний helper прямо у конкретному виконуваному .py, щоб після конвертації через PyInstaller не виникали проблеми з прихованими імпортами та залежностями.
   2. Якщо спільний модуль все ж потрібен, це має бути окреме свідоме рішення, а не автоматичний рефакторинг.

5. **Файл зайнятий іншим процесом**
   1. Якщо скрипт пише у файл, перед записом перевіряємо, чи файл не зайнятий. Для користувача така помилка має показуватись як зрозуміле вікно "Файл зайнятий", а не як технічні 'PermissionError', 'BadZipFile', 'repair', 'bad zip file' тощо.
   2. Типове повідомлення: файл використовується іншим процесом; закрийте цей файл або процес, що його використовує.
   3. Помилки доступу до файлу не ковтаємо всередині циклів обробки як звичайний warning/log. Якщо файл, у який треба писати, зайнятий — зупиняємо операцію і показуємо користувачу зрозумілу причину.

6. **Excel / openpyxl**
   1. Для '.xlsx' / '.xlsm' за замовчуванням використовуємо 'openpyxl', а не COM. COM використовуємо лише коли без нього справді неможливо виконати задачу, і це має бути явно обґрунтовано.
   2. Для '.xlsm' не забуваємо 'keep_vba=True'; для шаблонів, де важливі зв’язки, по можливості зберігаємо 'keep_links=True'.
   3. Збереження '.xlsx' краще робити через тимчасовий файл поруч із ціллю: 'wb.save(tmp_path)' -> перевірка валідності ZIP -> 'tmp_path.replace(target_xlsx)'. Це зменшує шанс отримати пошкоджений файл після збою.

7. **Excel-шаблони та форматування**
   1. Якщо потрібно "стерти текст", стираємо тільки 'cell.value', не чіпаючи 'fill', 'border', 'font', 'alignment', 'number_format' та інші стилі.
   2. Обережно з 'ws.cell(...)' та 'iter_rows(...)' під час очищення великих областей: вони можуть створювати порожні клітинки у файлі. Якщо треба очистити тільки вже існуючі клітинки шаблону, краще працювати з уже наявними клітинками.
   3. Не використовуємо 'delete_rows()' / 'delete_cols()' у складних Excel-шаблонах без крайньої потреби: це може пошкодити таблиці, формули, межі або внутрішню структуру книги. Для шаблонів краще імітувати очищення/зсув значень, зберігаючи структуру.

8. **Локальні шляхи та перевірка**
   1. Якщо відкритий у редакторі файл не знаходиться в поточній директорії, спершу перевіряємо дерево 'Codex' у Dropbox PYSCRIPTS, а не робимо висновок, що файл відсутній.
   2. Після змін у '.py' мінімальна перевірка — 'python -m py_compile <file.py>'. Якщо після перевірки створився '__pycache__', прибираємо його, щоб не залишати службові файли в робочому дереві.
